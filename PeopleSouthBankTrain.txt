```
{

  "total_credits_amount": <numeric value>,
  "total_debits_count": <integer number of debits>,

## STEP 3 — EXTRACT ALL TRANSACTIONS

Return a JSON array of every single transaction from "Activity in Date Order" section.
Include EVERY line — do NOT skip any transaction.

For each transaction, extract:

{
  "seq": <sequential integer starting at 1>,
  "date": "<date in MM/DD/YY format exactly as printed>",
  "description_line1": "<first line of description text>",
  "description_line2": "<second line if present, else null>",
  "description_line3": "<third line if present, else null>",
  "description_full": "<all description lines concatenated with space>",
  "transaction_type": "<one of: CREDIT | DEBIT | FEE>",
  "amount": <absolute numeric value — always positive>,
  "direction": "<CR for credit/deposit | DR for debit/withdrawal>",
  "running_balance": <numeric balance shown after this transaction>,
  "category": "<see Category Rules below>"
}

### AMOUNT PARSING RULES:
- If the amount column shows a number WITHOUT a minus sign → it is a CREDIT (money coming IN)
- If the amount column shows a number WITH a minus sign (e.g. 300.00-) → it is a DEBIT (money going OUT)
- The running balance column is the balance AFTER this transaction
- Always store amount as a positive number; use the "direction" field for CR/DR

### DESCRIPTION PARSING RULES:
This bank uses multi-line descriptions. Capture ALL lines that belong to one transaction.
Common multi-line patterns seen in this bank:
  - Line 1: Date + Transaction code + description
  - Line 2: Merchant name or additional detail
  - Line 3: Address or city/state
  - Line 4: Card number reference (e.g. C#2207, C#2189)

Example of a multi-line transaction — capture all lines as one record:
  1/07  POS DEB 1249 01/07/26 45081700    90.15-    27,348.82
        O REILLY 2223
        4620 HIGHWAY 90
        MARIANNA    FL  C#2207

This is ONE transaction, not four. description_full = "POS DEB 1249 01/07/26 45081700 O REILLY 2223 4620 HIGHWAY 90 MARIANNA FL C#2207"

### TRANSACTION CATEGORY RULES:
Assign one of these categories based on description keywords:

| Category              | Keywords / Patterns                                                    |
|-----------------------|------------------------------------------------------------------------|
| CARD_SALES_REVENUE    | NET SETLMT MERCH BANKCARD, CCD (when it's a credit)                   |
| MERCHANT_ADVANCE_IN   | RETRO ADVANCE INC, FUNDING METRICS LLC, advance credit deposits       |
| INTERNAL_TRANSFER_IN  | Transfer from x8914, Transfer from x5268, TRANSFER FROM x2189         |

| CHECK_WRITTEN         | CHECK followed by a check number                                      |

## STEP 6 — VALIDATION CHECKS

After extraction, perform these validations and report any failures:

1. BALANCE TRAIL CHECK:
   For every transaction, verify:
   previous_running_balance + credit_amount - debit_amount = current_running_balance
   (within $0.01 rounding tolerance)
   Flag any transaction where the balance does not reconcile.

2. TRANSACTION COUNT CHECK:
   Count your extracted transactions.
   Compare to "X Credits" and "Y Debits" shown in the account summary.
   credits_extracted + debits_extracted should equal total transactions.
   Flag if your count does not match.

3. TOTAL AMOUNT CHECK:
   Sum of all credit amounts = total_credits_amount from summary
   Sum of all debit amounts = total_debits_amount from summary
   Flag if sums don't match within $0.01.

4. DATE CONTINUITY CHECK:
   All transaction dates must fall within statement_period_start and statement_period_end.
   Flag any out-of-range dates.

5. RUNNING BALANCE CHECK:
   First transaction: previous_balance ± first_transaction_amount = first_running_balance
   Last transaction running_balance = current_balance
   Flag if these don't match.
```

## SECTION 3 — COMMON OCR ERROR PATTERNS & CORRECTIONS

### 3.1 NUMBER MISREADS
```
Watch for these common OCR errors in this statement format:

| OCR reads | Likely correct | Rule |
|-----------|---------------|------|
| "l,283.82" | 1,283.82 | lowercase L → digit 1 |
| "O REILLY" | O'REILLY   | letter O is correct here (merchant name) |
| "7l57.16"  | 7,157.16   | lowercase L → comma |
| ".00"      | 0.00       | leading dot = zero cents |
| "S 1,083.38" | $1,083.38 | S → $ sign |
| "22.55-SC" | 22.55 debit (service charge) | "-SC" suffix = debit fee |
| "3O"       | 30         | uppercase O → zero |

Thousands separator: comma (,)
Decimal separator:   period (.)
Negative indicator:  trailing minus sign "300.00-"
```

### 3.2 DESCRIPTION CONTINUATIONS
```
When a description wraps to the next line, do NOT create a new transaction.
A new transaction always starts with a DATE in the first column (MM/DD format).
Lines without a date in the first column are continuation lines of the previous transaction.

Example of how lines should be grouped:
  NEW TX:  "1/07  POS DEB 1249 01/07/26 45081700   90.15-   27,348.82"
  CONT:    "      O REILLY 2223"
  CONT:    "      4620 HIGHWAY 90"
  CONT:    "      MARIANNA    FL  C#2207"
  → These 4 lines = 1 transaction
  
  NEW TX:  "1/07  BILLNG    MERCH BANKCARD   45.17-   26,985.56"
  CONT:    "      CCD"
  → These 2 lines = 1 transaction

Special case — "CCD" or "PPD" alone on a line is always a continuation:
  "NET SETLMT MERCH BANKCARD"  followed by  "CCD"  = one transaction
```

### 3.3 AMOUNT COLUMN EDGE CASES
```
Edge case 1 — Zero interest:
  "Interest Paid    .00"  →  interest_paid = 0.00

Edge case 2 — Service charge with code:
  "1/30  Service Charge    22.55-SC    1,083.38"
  The "-SC" means it's a service charge debit. amount = 22.55, direction = DR

Edge case 3 — Small amounts:
  "3.66-"  →  debit of $3.66
  ".99-"   →  debit of $0.99

Edge case 4 — Large amounts with commas:
  "23,500.00"  →  credit of $23,500.00
  "30,000.00-" →  debit of $30,000.00

Edge case 5 — Overdraft fees:
  "Overdraft Item Fee    30.00-"
  These appear as line items AND in the summary totals. Count them in transactions only.
```

## SECTION 7 — POST-PROCESSING VALIDATION SCRIPT (Python)

```python
def validate_bank_statement(data: dict) -> dict:
    """
    Validates extracted bank statement data.
    Returns dict with validation results.
    """
    errors = []
    summary = data['account_summary']
    transactions = data['transactions']

    # 1. Balance equation check
    calc_balance = (
        summary['previous_balance']
        + summary['total_credits_amount']
        - summary['total_debits_amount']
        - summary['service_charge']
    )
    if abs(calc_balance - summary['current_balance']) > 0.01:
        errors.append(
            f"BALANCE MISMATCH: calculated={calc_balance:.2f}, "
            f"reported={summary['current_balance']:.2f}"
        )

    # 2. Transaction count check
    credits = [t for t in transactions if t['direction'] == 'CR']
    debits  = [t for t in transactions if t['direction'] == 'DR']
    if len(credits) != summary['total_credits_count']:
        errors.append(
            f"CREDIT COUNT: extracted={len(credits)}, "
            f"reported={summary['total_credits_count']}"
        )
    if len(debits) != summary['total_debits_count']:
        errors.append(
            f"DEBIT COUNT: extracted={len(debits)}, "
            f"reported={summary['total_debits_count']}"
        )

    # 3. Running balance trail check
    for i, tx in enumerate(transactions):
        if i == 0:
            prev_bal = summary['previous_balance']
        else:
            prev_bal = transactions[i-1]['running_balance']

        if tx['direction'] == 'CR':
            expected = prev_bal + tx['amount']
        else:
            expected = prev_bal - tx['amount']

        if abs(expected - tx['running_balance']) > 0.01:
            errors.append(
                f"BALANCE TRAIL ERROR at seq {tx['seq']} "
                f"({tx['date']} {tx['description_full'][:40]}): "
                f"expected={expected:.2f}, got={tx['running_balance']:.2f}"
            )

    # 4. Credit/debit total sums
    credit_sum = sum(t['amount'] for t in credits)
    debit_sum  = sum(t['amount'] for t in debits)
    if abs(credit_sum - summary['total_credits_amount']) > 0.01:
        errors.append(
            f"CREDIT SUM: extracted={credit_sum:.2f}, "
            f"reported={summary['total_credits_amount']:.2f}"
        )
    if abs(debit_sum - summary['total_debits_amount']) > 0.01:
        errors.append(
            f"DEBIT SUM: extracted={debit_sum:.2f}, "
            f"reported={summary['total_debits_amount']:.2f}"
        )

    return {
        "valid": len(errors) == 0,
        "error_count": len(errors),
        "errors": errors
    }
```
